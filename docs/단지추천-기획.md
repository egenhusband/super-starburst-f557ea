## 1. 한 줄 요약

> 대출 계산 결과 화면에서, **입력한 주택가격 기준으로 둘러볼 만한 수도권 단지**를 한 번의 탭으로 보여준다.

계산기를 **"한도를 알려주는 도구"**에서 **"의사결정을 돕는 도구"**로 확장한다.

---

## 2. 배경 · 문제 정의

- 현재 서비스는 대출 한도를 계산해주지만 결과가 **"4.2억 가능"에서 끝난다.** 사용자의 다음 질문("그래서 어디 살 수 있는데?")으로 이어지지 않는다.
- 서비스는 이미 **단지 실거래가, 입지 등급([analyze-apt.js](../netlify/functions/analyze-apt.js)), 초등학교 거리·세대수 데이터**를 보유. **추천에 필요한 재료가 이미 있다.**
- 신규 데이터 수집 없이 **기존 자산을 연결**하는 것만으로 차별화 기능을 만들 수 있다.

## 4. 핵심 컨셉 — "입력한 주택가격 기준"

계산기에서 사용자가 입력한 주택가격을 추천 가격 상한으로 사용한다. 새로 물어볼 입력이 거의 없다.

> ⚠️ **용어**: "살 수 있는 단지"가 아니라 **"입력한 주택가격 기준으로 둘러볼 단지"**로 표기한다. 다른 가격대는 LTV·DTI·방공제 한도가 달라질 수 있으므로 단정하지 않는다.

### 4-1. 추천 가격 상한

```
recommendPriceCap = inputPrice
```
- `inputPrice`: 계산기 주택가격 입력값
- 입력 가격보다 높은 단지는 기본 추천에 포함하지 않는다.

### 4-2. 왜 자산+대출한도를 추천 상한으로 쓰지 않는가

계산 결과의 대출 한도는 **입력한 주택가격 기준**으로 산출된다. 추천 단지 가격이 입력가보다 높아지면 LTV·DTI·방공제·상품 자격이 다시 달라질 수 있다.

따라서 MVP에서는 `자산 + 대출한도`를 새로운 구매 상한으로 확장하지 않는다. 더 높은 가격대를 보고 싶다면 별도 재계산 UX가 필요하다.

### 4-3. 기본 지역

추천 기본값은 항상 **수도권 전체**다. 계산기의 `region`은 대출 규제지역 입력이므로 사용자가 원하는 거주지역으로 해석하지 않는다. 지역은 추천 화면 드롭다운에서 직접 좁힌다.

## 5. 범위 (Scope) — 수도권 한정 (결정사항)

### 왜 수도권만?

데이터 실측 결과 **교통·입지 등급 신뢰도가 수도권에 집중**. 전국으로 펼치면 지방의 빈약한 데이터가 노출되어 신뢰를 깎는다. "전국 되는 척"보다 "수도권 정직하게".

### 데이터 커버리지 실측 (2026-06-01)

| 데이터 | 보유 단지 | 전국 대비 | 추천 활용 |
|---|---:|---:|---|
| 전체 단지 (`apt-code-map.json`) | 22,132 | 100% | 기준 모집단 |
| 실거래가 (`apt-area-prices/`) | 10,679 | ~48% | ✅ 가격 (단, **지방 포함 → 빌드 시 필터 필수**) |
| 초등학교 거리 (`apt-school-meta`) | 7,465 | ~34% | ✅ 초등 도보권 |
| **역세권/교통 (`apt-station-meta`)** | **208** | **~1%** | ⚠️ MVP 제외 (2차) |
| 지하철 소요시간 그래프 | 서울 | — | ⚠️ MVP 제외 (2차) |

> [analyze-apt의 getLocationTier](../netlify/functions/analyze-apt.js#L310)는 서울·경기만 명시적 등급을 매기고 나머지는 전부 T5로 일괄 처리 → 지방은 등급 변별력 없음.

> ⚠️ **중요 (빌드 정확성)**: 원본 `apt-code-map.items`에는 **`regionKey`가 없다.** `as1` 필드만 있으며 값은 `"서울특별시"`/`"경기도"` 등 시도 풀네임이다.
> → 추천 인덱스 빌드 시 **`as1 === "서울특별시" || as1 === "경기도"`로 선별**하고, 그때 `regionKey`(`seoul`/`gyeonggi`)를 **새로 생성**해 인덱스에 넣는다. (`apt-area-prices/`에 지방 단지가 섞여 있으므로 이 필터가 반드시 선행)

### 향후 확장 조건

지방 확장은 `apt-station-meta` 및 지역별 지하철 그래프 보강 후 3차.

## 7. 화면 명세

### 7-1. 진입점 — 결과화면 CTA 카드

- 위치: 결과 화면([data-slide="8"](../index.html#L328) `resultContent`)의 상품 정보 그룹 바로 아래, 한도 상세 카드 위
- 구성: 🏘️ + 타이틀 + 기준 문구(`입력한 주택가격 5.0억 이하 · 평균 실거래가 기준`)
- **노출 조건 (확정)**: 받을 수 있는 대출 상품이 **1개 이상일 때만** CTA 표시. 자격 미달이면 CTA 대신 "조건을 바꿔 다시 계산" 안내.
- 동작: `startRecommendFromResult({ targetPrice, region: 'all' })` 호출 → 리스트 진입
- **유료 정책 연동** → §11 참조

### 7-2. 추천 리스트 화면 (단일 화면)

**상단 고정 필터바**

| 행 | 요소 | 내용 |
|---|---|---|

| 1행 | 가격 슬라이더 | 계산기에 입력한 주택가격 이하에서 조정 |

**단지 카드 — 예산 맞춤 평형 매칭 (v3 변경)**

```
🥇 광교 더샵 레이크파크          [S]
   경기 수원 영통                       84㎡ · 입력가 대비 0.3억 낮음
   1,230세대  2019년  초등 도보권 280m  평균 5.9억 · 최근 6.1억(2026-05)
```

- 좌: 등급 배지(S/A/B, **인덱스에 사전계산된 값**)

- 탭 → 기존 단지 상세 화면

**빈 상태**: "조건에 맞는 단지를 못 찾았어요…" + [입력가 상한으로 다시 보기]

## 8. 추천 · 랭킹 로직

### 8-1. 필터 — 평형 단위 매칭 (v3 변경)

```
1차: 단지의 areas 중 최소 1개 평형의 avgPrice ≤ 계산기에 입력한 주택가격
     → 매칭된 평형 중 "입력가 이하 최대 평형"을 카드에 표시 (표시가는 latestPrice+latestDate 병기)
2차: regionKey ∈ {seoul, gyeonggi} (+ 사용자가 좁힌 시군구)
```

> 단일 대표가로 거르지 않는다. 단지 안에서 **입력한 주택가격 이하의 평형이 하나라도 있으면 노출**하고, 그중 가장 큰 평형을 보여준다. 다른 가격대는 대출 한도가 달라질 수 있으므로 기본 추천 범위에 포함하지 않는다.

### 8-2. 정렬 (MVP)

| 칩 | 정렬 기준 | 인덱스 필드 |
|---|---|---|
| 종합순(기본) | 등급 점수 desc | `clampedScore` |
| 초등 도보권 | 초등학교 거리 asc | `schoolDistance` |
| 신축 | 준공연도 desc | `buildYear` |
| 대단지 | 세대수 desc | `householdCount` |
| (2차) 직주근접/역세권 | 소요시간/역거리 | `stationDistance` 등 |

### 8-3. 핵심 원칙 — analyze-apt는 런타임에서 호출하지 않는다 (검토 반영)

> 후보가 수천 개일 때 단지마다 analyze-apt API를 호출하면 안 된다.
> **등급·점수·거리 등을 빌드 시 사전계산해 인덱스에 적재**하고, 런타임은 정적 인덱스를 받아 **클라이언트에서 즉시 필터·정렬**한다.

## 9. 데이터 정직성 원칙 ⭐

- 면책 문구는 가격 기준·시세·등급 각각에 명시 (§12).

## 10. 구현 방향 (검토 반영)

### 10-1. 신규 인덱스 생성

- **스크립트**: `scripts/update-apt-recommend-index.js` (기존 `scripts/update-apt-*.js` 패턴 재사용)
- **출력**: `data/apt-recommend-index.json` (← `apt-trades-summary.json` 재사용하지 않음)
- **빌드 절차**:
  1. `apt-code-map.items`에서 **`as1 === "서울특별시" || "경기도"`로 선별** → 각 단지에 `regionKey`(`seoul`/`gyeonggi`) 생성
  2. 각 단지의 `byArea`(면적별 실거래가)를 **압축 형태로 적재** — `price`는 **`avgPrice`**, 표시 보조로 `latestPrice`/`latestDate` 동반 (단일 대표가 산정 X)
  3. `apt-school-meta.entries` / `apt-station-meta.entries` (**배열**) 및 `apt-households`를 **`kaptCode` 기준 Map으로 만들어 조인** (`Object.keys(json)` 사용 금지)
  4. analyze-apt 로직(`computeAptGrade`)을 **빌드 타임에 1회 실행**해 `grade`, `clampedScore` 적재

### 10-2. 인덱스 필드 (v3 변경)

```
kaptCode, aptName, sigunguName, umdName, regionKey,
areas: [ { area:"84㎡", avgPrice:61000, latestPrice:63000, latestDate:"2026-05-20" }, ... ],  ← 면적별 실거래가 묶음
minAvgPrice, maxAvgPrice,                                              ← 빠른 필터용 보조 인덱스(avgPrice 기준)
grade, clampedScore,
schoolDistance, stationDistance, householdCount, buildYear
```

- **필터/정렬 기준 = `avgPrice`** (안정적). **표시 = `latestPrice` + `latestDate` 병기** (신선도).
  - 예: "84㎡ 평균 6.1억 · 최근 6.3억(2026-05)"
- `minAvgPrice`/`maxAvgPrice`는 "입력 가격에 단 하나도 안 들어오는 단지"를 1차로 빠르게 거르기 위한 보조값.
- 용량: 수도권 ~8,983개 단지 × 평형 수. 단지당 평형 평균 3~5개 → 단일 JSON 수 MB 수준, 지연 로드(lazy) 처리.

### 10-3. 결과화면 CTA

- 신규 함수 `startRecommendFromResult()`
- 넘기는 값: **`targetPrice`, `region: 'all'`** 중심. 계산기의 `region`은 대출 규제지역 성격이므로 추천 기본 지역에는 반영하지 않는다.

### 10-4. 추천 리스트

- 클라이언트가 `apt-recommend-index.json` 로드 → 즉시 필터/정렬 (서버 호출 없음)
- 단지 상세 진입 시 **기존 단지 상세 화면 재사용**

### 10-5. 재사용 / 신규 정리

| 재사용 | 신규 |
|---|---|
| 단지 상세·등급 화면, region/sigungu 데이터, 디자인 토큰, computeAptGrade(빌드용) | CTA 카드+함수, 추천 리스트 화면, `update-apt-recommend-index.js`, `apt-recommend-index.json` |

  - 선택 이유: 탐색은 자유롭게 열어 가치를 체감시키고, 깊은 정보(상세)에서 기존 과금 흐름을 그대로 재사용.
